# Quota Radar 布局矩阵稳定性设计

## 背景

Quota Radar 的主界面由三组可组合状态决定：

- Provider 内容：仅圆环、圆环加卡片，以及两个 Provider 不同状态的混合模式。
- Provider 排列：上下或左右。
- 布局尺寸：紧凑、标准或宽松。

现有实现把 Provider 排列、面板内部排列、滚动轴和窗口自动缩放耦合在一起。结果包括宽松模式横向裁切、左右模式窗口伸出屏幕、双 Provider 高度依赖切换历史、混合卡片状态失去尺寸适配，以及单 Provider 仍按双列宽度计算。

## 目标

1. 12 个主组合和混合卡片状态都不产生不可访问的裁切。
2. 窗口自动调整后始终位于当前屏幕的可见区域。
3. Provider 外层排列不再决定单个面板内部的圆环/卡片排列。
4. 尺寸策略基于实际渲染内容，而不是仅基于设置中勾选的卡片 ID。
5. 双 Provider 左右仅圆环时收起多余高度；上下排列时至少完整显示首个面板。
6. 0、1、2 个可见 Provider 都有明确结果。
7. 布局规则可通过纯单元测试覆盖，并用真实 `.app` 回归关键切换序列。

## 方案比较

### 方案 A：在 `ContentView` 中继续追加条件分支

直接修改滚动轴、最小宽度和窗口 resize 条件。改动较小，但状态组合继续散落在视图代码中，混合状态和后续 Provider 扩展仍容易产生新回归。

### 方案 B：提取纯布局策略，保留窄 `NSWindow` 桥接（采用）

新增可测试的 `DashboardLayoutPolicy`。它接收 preset、Provider 排列、实际可见 Provider 和每个 Provider 是否有实际卡片，输出：

- 外层滚动轴需求；
- 每个 Provider 的建议/最小宽度；
- 窗口建议宽高和最低可用视口；
- 是否应自动适配宽度或高度。

SwiftUI 只负责渲染策略结果，现有 `NSViewRepresentable` 只负责把目标尺寸约束到 `screen.visibleFrame`。这种方案改动集中、可测试，并兼容当前 macOS 14 部署目标。

### 方案 C：完全改成 SwiftUI scene placement API

使用更新的窗口 placement/ideal placement API 重构窗口生命周期。API 更现代，但项目部署目标和动态切换需求仍需要 AppKit 兜底，且会扩大本次改动范围，不适合作为 v1.0.4 维护修复。

## 设计

### 1. 布局输入模型

为每个可见 Provider 生成 `ProviderLayoutContent`：

- `provider`
- `hasRenderedCards`

`hasRenderedCards` 由当前 snapshot 能实际生成的卡片决定。加载、失败、订阅卡缺失或仅勾选无数据卡片时，如果没有实际卡片，按仅圆环处理。

### 2. 外层与面板内部解耦

`ProviderLayoutMode` 只决定 Provider stack：

- `vertical`：Provider 使用 `VStack`。
- `horizontal`：Provider 使用 `HStack`。

单个 `ProviderPanelView` 不再读取外层排列。只要有卡片，统一使用 `ViewThatFits(in: .horizontal)`：宽度足够时圆环和卡片左右排列，否则圆环在上、卡片在下。仅圆环时保持居中方形面板。

### 3. 宽度和滚动规则

- 仅圆环面板使用 preset 对应的方形宽度。
- 有卡片面板的最低宽度允许一列卡片完整显示；建议宽度优先容纳两列。
- 单 Provider 使用全部可用宽度，不再固定除以 2。
- 左右排列根据实际 Provider 数量分配宽度。
- 当窗口小于组合内容的最低宽度时启用横向滚动，任何外层排列都不能产生不可访问裁切。
- 动态 `contentMinSize` 至少保证标题、完整圆环或一列卡片可达。

### 4. 高度规则

- 左右＋全部仅圆环：窗口高度适配最高面板。
- 上下＋全部仅圆环：窗口至少完整显示第一个面板；屏幕可容纳时显示全部。
- 只有一个 Provider：高度适配该 Provider 的内容并约束到屏幕。
- 有卡片时保留用户高度，但最低视口必须容纳标题和完整圆环；其余内容通过纵向滚动访问。
- 从一种组合切换到另一种组合时，结果不依赖此前是否手工缩放过窗口。

### 5. 屏幕边界

窗口目标 frame 使用当前 `screen.visibleFrame` 约束：

- 宽高不超过可见区域。
- 扩宽时若右侧空间不足，向左移动。
- 扩高时保留顶部位置，但不越过 Dock 或屏幕底部。
- 目标内容大于屏幕时，窗口取可见区域允许的最大尺寸，并依靠滚动访问内容。

### 6. 空状态

两个 Provider 都关闭时显示明确空状态：说明当前未显示 Provider，并提供打开设置的按钮。窗口保持正常 Dock App 行为和工具栏。

### 7. 测试策略

先为纯布局策略编写失败测试，再实现：

- 12 个主组合。
- 两个方向的混合卡片状态。
- 0、1、2 个可见 Provider。
- 勾选卡片但没有实际卡片数据。
- 紧凑、标准、宽松的最小/建议尺寸。
- 窄窗口到宽松、上下到左右、有卡片到无卡片的切换。
- 目标 frame 始终包含于 `visibleFrame`。

最后运行全部 Swift 测试，并用实际 `.app` 检查以下序列：

1. 紧凑＋上下＋仅圆环 → 宽松。
2. 紧凑窄窗口＋卡片 → 左右。
3. 左右＋仅圆环在三种尺寸间切换。
4. 一个 Provider 有卡片、另一个仅圆环。
5. 单 Provider 和无 Provider。

## 非目标

- 不改变额度数据来源、解析或刷新逻辑。
- 不重新设计配色、卡片视觉或信息层级。
- 不改变默认设置值。
- 不引入仅适用于 macOS 15 以上的新窗口 API。

## 完成标准

- 上述自动化测试全部通过。
- 真实 App 中所有关键序列没有不可访问裁切、屏外窗口或明显无意义空白。
- 外层 Provider 排列和内部面板响应式排列行为可独立解释。
- 源码、`DESIGN.md`、测试和 `.planning` 交接记录一致。
